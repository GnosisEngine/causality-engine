Fief Graph Engine

Technical Specification

This document specifies the entity-verb-entity (EVE) graph engine underlying the Fief MMORPG. The engine replaces conventional stateful engineering solutions — subscription cascades, ECS cache invalidation, explicit cycle detection, temporal indexing — with stateless mathematical primitives whose correctness properties are intrinsic rather than enforced. The graph is the authoritative source of relational truth. All other systems are derived projections.

The engine is designed for epistemic slipperiness by default. Relational truth is continuous, not binary. Agent knowledge of that truth is partial, delayed, and uncertain by construction. These are features of the model, not defects to be corrected.

01

Introduction & Guiding Principles

Design axioms

Every architectural decision traces back to one of the following axioms. When trade-offs arise, the lower-numbered axiom takes precedence.

#AxiomImplication
A1The graph is the source of truthNo system may hold authoritative relational state outside the graph. ECS and other caches are projections.
A2Math over machineryWhere a mathematical property ensures correctness, no runtime enforcement mechanism is needed or desired.
A3Epistemic limits are designPlayers and NPCs having partial, delayed, incorrect knowledge of the world is correct behavior, not a deficiency.
A4Uncertainty is a valueRelational state is a continuous quantity with associated confidence. Binary forced answers are a presentation-layer concern.
A5Seedable determinismAll graph evolution must be reproducible from an event log and a seed. Nondeterministic external state is prohibited in write paths.

What this spec covers

Data structures for all graph primitives. Mathematical primitive interfaces and their runtime contracts. Multi-graph federation model. Query interface shape. Persistence and replay protocol. Node.js-specific runtime constraints.

What this spec defers

Fluent query DSL syntax. Specific game-domain verb taxonomies. Rendering and UI. Auth and session management. Content tooling for designers.

02

System Architecture

The engine consists of five layers. Each layer depends only on layers below it. No upward dependencies are permitted.

LayerNameResponsibility
L5Query InterfaceTraversal, scoring, nearest-neighbor, diffusion evaluation. Async-safe.
L4Projection CacheECS-compatible derived values. Dirty versioning. Worker thread pool.
L3Math PrimitivesFuzzy composition, embedding ops, diffusion, DS belief, HDC bundle.
L2Graph CoreEntity and edge storage. Event log. Meta-graph. Write validation.
L1PersistenceEvent log flush. Snapshot checkpoints. Graph ID registry.

Graph instances in play

InstanceMutableTemporalIn-memory policyPrimary owner
static-rulesNoNoAlways fully loadedServer boot
dynamic-worldYesOptionalHot zones in memory, cold pagedZone servers
politicalYesYesFully loaded, persisted with historyGlobal server

Additional graph instances are permitted. Each must declare its characteristics at registration. Cross-instance references follow the boundary verb protocol defined in §6.

Data flow

Game systems write to the graph via the event log (L2). The graph validates writes against the meta-graph and active inference rules. On commit, the math primitives layer updates affected embeddings and belief bundles. The projection cache marks dependent components dirty. Query callers read either directly from the graph or from the projection cache depending on their latency requirements.

Core Data Structures

3.1 EntityNode

The fundamental node type. All entities in all graph instances conform to this structure. Type-specific extension data lives in properties.

/**
 * @typedef {Object} EntityNode
 */
{
  id:            string,         // UUID v4. Globally unique across all graphs.
  graphId:       string,         // Owning graph instance id.
  type:          string,         // Registered entity type from type registry.
  properties:    Map<string,any>,// Type-specific extension data. Serialisable.

  // KG embedding — see §5.3
  embedding:     Float32Array,   // Length = GraphInstance.embeddingDim.
  embeddingVer:  number,         // Incremented on each embedding update.

  // HDC memory bundle — see §5.6. Null if instance does not use HDC.
  memoryBundle:  Int8Array | null,// Length = GraphInstance.hdcDim.

  // Lifecycle
  logSeqNum:     number,         // Event log sequence number at creation.
  version:       number,         // Incremented on any neighborhood change.
}

3.2 VerbEdge

The fundamental edge type. All edges are directed. Bidirectional relationships are modeled as two edges, potentially with asymmetric weights. Inferred edges are distinguished from asserted edges by authority.

/**
 * @typedef {Object} VerbEdge
 */
{
  id:            string,         // UUID v4.
  sourceId:      string,         // EntityNode id. May reference a foreign graph.
  targetId:      string,         // EntityNode id. May reference a foreign graph.
  verbType:      string,         // Registered verb from meta-graph. See §4.

  // Fuzzy membership — see §5.1
  weight:        number,         // [0, 1]. Fuzzy membership value.
  tNorm:         TNormId,        // Composition rule. Inherited from MetaVerb if null.

  // Epistemic provenance — see §5.5
  authority:     Authority,      // 'system' | 'inference' | 'player' | 'world'
  confidence:    number,         // [0, 1]. Asserted edges default to 1.0.
  assertedBy:    string,         // Entity id of the asserting agent. Null for system.

  // Optional extension
  properties:    Map<string,any>,// Arbitrary metadata. Must be serialisable.
  expiresAt:     number | null,   // Log sequence number at which this edge is void.

  // Lifecycle
  graphId:       string,         // Owning graph. May differ from source/target graphs.
  logSeqNum:     number,         // Event log seq at creation.
}
Invariants

weight is always in [0, 1]. Values outside this range are rejected at write time.

confidence is always in [0, 1]. Inferred edges must have confidence < 1.0.

A VerbEdge whose verbType is a boundary verb (see §6.2) must have sourceId and targetId in different graph instances.

expiresAt, if set, must be greater than the current log sequence number at write time.

3.3 GraphInstance

Configuration object for a graph instance. Registered at boot. Immutable after registration.

{
  id:              string,          // Unique short identifier, e.g. 'static-rules'.
  label:           string,

  // Characteristics
  mutable:         boolean,
  temporalIndex:   boolean,         // If true, mutations append; current = materialized view.
  persistenceMode: PersistenceMode, // 'full' | 'log-only' | 'ephemeral'

  // Math primitive configuration
  embeddingDim:    number,          // KG embedding dimensionality. Default: 128.
  hdcDim:          number | null,   // HDC bundle size. Null disables HDC for this graph.
  defaultTNorm:    TNormId,         // Fallback t-norm for edges without explicit tNorm.
  epsilon:         number,          // Inference termination threshold. Default: 0.001.

  // Memory management
  hotZoneSelector: Function | null, // Returns true for entities to keep in memory.
  pageSize:        number,          // Entities per page for cold storage. Default: 512.
}

3.4 EventLogEntry

Every mutation to the graph — including inferred edge updates — is recorded as an event log entry. The log is the source of truth for persistence and replay. No mutation occurs without a log entry.

{
  seq:        number,           // Monotonically increasing. Global across all graphs.
  graphId:    string,
  type:       EventType,        // See event type table below.
  payload:    Object,           // Type-specific. Must be fully serialisable.
  rngState:   string,           // PRNG state snapshot at write time for replay.
  wallMs:     number,           // Wall-clock ms. For observability only; not used in replay.
}

edge:addVerbEdge (full)
edge:updateid, graphId, delta (changed fields only)

04

Meta-Graph — Verb Taxonomy

The meta-graph is a second graph whose nodes are verb types and whose edges are ontological relationships between them. It lives entirely in memory, is loaded at boot, and is treated as read-only during server operation unless a rule registration transaction is in progress.

4.1 MetaVerb

{
  id:               string,          // Unique verb type identifier, e.g. 'isAlliedWith'.
  label:            string,
  domain:           string[],        // Permitted source entity types.
  range:            string[],        // Permitted target entity types.
  defaultTNorm:     TNormId,         // Composition t-norm for this verb type.
  defaultPolicy:    PolicyFnId,      // Convergence policy for transitive rules.
  inferenceStratum: number,          // Assigned at registration by stratification.
  symmetric:        boolean,         // If true, assert A→B also asserts B→A.
  crossGraphAllowed:boolean,         // May this verb type cross graph boundaries?
}

4.2 Taxonomy edge types

Taxonomy verbSemanticsEffect on queries
isSubtypeOfA is a specialization of BQueries for B match A edges automatically
isInverseOfA's target→source is BAsserting A auto-asserts B in reverse direction
impliesPresence of A with weight w implies B with w × factorInference rule registered automatically
excludesA and B cannot coexist above threshold on same pairWrite validation rejects or overwrites conflicting edge
requiresPolicyTransitive rules on A must use named policy functionPolicy validation at rule registration

4.3 Registration and validation

MetaVerb registration is a transaction. On commit, the system runs a topological sort over all inference rules, assigning strata. If a cycle through negation is detected, the transaction is rolled back and the error is reported with the offending rule chain identified. No runtime behavior is affected.

Registration invariants

Every transitive inference rule must name a registered PolicyFn that the engine can verify converges to zero.

No two MetaVerb registrations may share an id within the same graph instance.

A MetaVerb with crossGraphAllowed: false may not appear in the boundary verb registry (§6.2).

Symmetric verbs with an explicit isInverseOf edge are rejected (self-contradictory).

05.1

Fuzzy Graph Theory — Relational Truth

All edge weights are fuzzy membership values in [0, 1]. Composition of relationships along a path uses a t-norm, which must be selected per verb type at registration. The t-norm is the primary lever for how relationship strength propagates across hops.

T-norm implementations

TNormIdFormulaSemanticsUse case
minimummin(a, b)Chain strength = weakest linkAuthority chains, delegation
producta × bStrength erodes with each hopTrust, political influence
lukasiewiczmax(0, a+b−1)Aggressive decay, zero floorFear, debt, coercion
drasticmin(a,b) if max=1, else 0Requires one perfect linkDirect kinship, treaty obligations

Composition along a path

Definition — path weight
w(path) = tNorm(w₁, tNorm(w₂, tNorm(w₃, ... wₙ))) where wᵢ is the weight of the i-th edge and tNorm is the verb's registered t-norm. A path weight below GraphInstance.epsilon is treated as non-existent.

Exclusion handling

When a write would create an edge whose verbType excludes an existing edge on the same (source, target) pair, the resolution is determined by the excludes taxonomy edge's resolutionPolicy property: reject rejects the new write; overwrite removes the conflicting edge; weaken reduces both weights by the incoming weight value. If no resolutionPolicy is set, reject is the default.

05.2

Convergent Policy Functions — Inference

Every inference rule that operates transitively must supply a policy function. The policy function takes a weight and a hop count and returns a new weight. The engine verifies at registration time that the function converges to zero.

Interface

/**
 * A policy function that controls transitive inference weight decay.
 * MUST satisfy: lim(hop → ∞) apply(w₀, hop) = 0 for all w₀ ∈ (0, 1].
 *
 * @typedef {Object} PolicyFn
 */
{
  id:          string,
  apply:       (weight: number, hop: number) => number,
  epsilon:     number,    // Weight below which inference halts. Default: inherit from graph.
  verify:      (w0: number, maxHops?: number) => boolean,
                           // Engine calls this at registration. Must return true.
}

Built-in policy functions

// Geometric decay. Most common case.
const geometric = (discount) => ({
  id: `geometric(${discount})`,
  apply: (w, hop) => w * Math.pow(discount, hop),
  epsilon: 0.001,
  verify: () => discount > 0 && discount < 1,
})

// Harmonic decay. Slower fall-off than geometric, useful for cultural influence.
const harmonic = ({
  id: 'harmonic',
  apply: (w, hop) => w / (hop + 1),
  epsilon: 0.001,
  verify: () => true,
})

// Threshold cutoff. Converges immediately after N hops.
// Use when you want explicit hop limits without fighting the model.
const cutoff = (maxHops) => ({
  id: `cutoff(${maxHops})`,
  apply: (w, hop) => hop >= maxHops ? 0 : w,
  epsilon: 0.001,
  verify: () => maxHops > 0,
})

Verification procedure

At registration, the engine calls verify() and additionally runs a simulation: starting at weight 1.0, it applies the policy function for up to 10,000 hops, confirming the output crosses epsilon before termination. Functions that do not cross epsilon within 10,000 hops are rejected. This simulation is cheap and runs once at boot.

05.3

KG Embeddings — State Projection

Each entity holds an embedding vector in a continuous space of dimension GraphInstance.embeddingDim. Relational state between two entities is a geometric relationship in this space rather than a boolean flag. The projection cache reads embedding distances rather than derived ECS values wherever the latency budget allows.

TransE update rule

Definition — TransE score
score(h, r, t) = − ‖h + r − t‖₂ where h = embedding of source entity r = embedding of verb type (per-verb vector, stored in MetaVerb) t = embedding of target entity Higher score = stronger evidence the (h, r, t) triple is true. An edge with weight w contributes: update t += α × w × (h + r − t) where α is the learning rate (default: 0.01).

Embedding operations

/**
 * Updates embeddings for source, verb, and target on edge write.
 * Called synchronously during graph commit, before log flush.
 */
function updateEmbedding(edge: VerbEdge, graph: GraphInstance): void

/**
 * Returns score in [0, 1] (normalised from TransE distance).
 * Does not traverse the graph. O(d) where d = embeddingDim.
 */
function score(sourceId, verbType, targetId, graph): number

/**
 * Returns k entities whose embeddings are nearest to
 * (source.embedding + verbType.embedding).
 * Used for "entities in a similar relational position" queries.
 * O(n × d) naive; use HNSW index for n > 10,000.
 */
function nearest(sourceId, verbType, k, graph): EntityNode[]
Performance notes

Embedding updates are synchronous during commit. Keep embeddingDim ≤ 256 for the dynamic-world graph to maintain acceptable commit latency on Node.js's main thread.

The static-rules graph, being immutable, has its embeddings computed once at boot and never updated.

For graphs with more than 50,000 entities, delegate nearest() to a worker thread with an HNSW index.

05.4

Influence Diffusion — Change Propagation

Political belief and reputation evolution are modeled using the DeGroot influence diffusion model. Systems query the current diffused state on demand rather than registering for change notifications. The diffusion computation is the propagation model. No observer pattern is required for slow-changing relational state.

DeGroot model

Definition — DeGroot update
b(t+1) = T · b(t) where b(t) = belief vector over all entities at step t T = trust/influence matrix derived from edge weights Tᵢⱼ = weight of edge (j → i) of the relevant influence verb type Friedkin-Johnsen extension with stubbornness parameter λᵢ ∈ [0, 1]: b(t+1)ᵢ = λᵢ · b(0)ᵢ + (1 − λᵢ) · Σⱼ Tᵢⱼ · b(t)ⱼ λᵢ = 1.0 means entity i ignores all external influence (fully stubborn). λᵢ = 0.0 means entity i is fully persuadable (no anchor).

Interface

/**
 * @typedef {Object} DiffusionConfig
 */
{
  verbType:      string,   // Which edge type forms the influence matrix.
  maxSteps:      number,   // Iteration cap. Default: 100.
  convergeDelta: number,   // Stop when max belief change per step < this. Default: 0.001.
}

/**
 * Evaluates DeGroot diffusion from a given starting entity.
 * Returns belief values for reachable entities after convergence.
 * Runs in a worker thread for graphs with > 1000 entities.
 */
async function diffuse(
  startId:   string,
  config:    DiffusionConfig,
  graph:     GraphInstance
): Promise<Map<string, number>>   // entityId → converged belief

Stubbornness as entity property

The stubbornness parameter λ is stored in EntityNode.properties.stubbornness as a number in [0, 1]. Default is 0.5 if not set. Game systems may modify stubbornness through normal graph writes — it does not require a special API.

Convergence rate as world texture

The second-largest eigenvalue of the trust matrix T determines how quickly beliefs homogenize. A value near 0 means rapid convergence; near 1 means sustained divergence. Designers can tune this indirectly by adjusting base edge weights and stubbornness defaults per entity type, creating political landscapes with different levels of ideological rigidity without any additional code.

05.5

Dempster-Shafer — Agent Epistemic State

Each agent capable of holding beliefs maintains a Dempster-Shafer belief function over graph propositions of interest. This is distinct from the graph's objective state. An agent's belief function represents what it knows, suspects, and is ignorant of — including the distinction between "I believe this is false" and "I have no information."

Belief function structure

/**
 * A basic probability assignment (BPA) over a two-element frame
 * {true, false} for a specific proposition.
 *
 * @typedef {Object} BeliefFunction
 */
{
  proposition: string,    // Serialised graph pattern, e.g. '(A,isAlliedWith,B)'.
  mTrue:       number,    // Mass assigned to {true}. Evidence for.
  mFalse:      number,    // Mass assigned to {false}. Evidence against.
  mUnknown:    number,    // Mass assigned to {true, false}. Explicit ignorance.
                           // Invariant: mTrue + mFalse + mUnknown = 1.0
  bel:         number,    // Derived. Belief lower bound = mTrue.
  pl:          number,    // Derived. Plausibility upper bound = mTrue + mUnknown.
  updatedAt:   number,    // Log seq of last update to this belief.
}

Dempster's rule of combination

Combining two belief functions over the same proposition
m₁₂({true}) = (m₁(T)·m₂(T) + m₁(T)·m₂(Ω) + m₁(Ω)·m₂(T)) / K m₁₂({false}) = (m₁(F)·m₂(F) + m₁(F)·m₂(Ω) + m₁(Ω)·m₂(F)) / K m₁₂({Ω}) = (m₁(Ω)·m₂(Ω)) / K where K = 1 − (m₁(T)·m₂(F) + m₁(F)·m₂(T)) [normalisation, conflict term] Ω = {true, false} [the ignorance set] If K = 0, the two sources are in total conflict. Resolution policy: retain higher-authority source.

Belief propagation across graph boundary

When a boundary edge delivers an update (after its temporal offset elapses), the receiving agent's belief function is updated via Dempster's rule using the incoming evidence, discounted by the boundary verb's transmissionFidelity property (a number in [0, 1] that adds mass to mUnknown proportional to 1 − fidelity). Low-fidelity boundary verbs represent rumour, second-hand intelligence, or corrupted transmission. High-fidelity verbs represent direct observation or formal treaty notification.

05.6

Hyperdimensional Computing — Agent Memory

NPC relational memory is modeled as a hyperdimensional bundle: a single high-dimensional vector formed by superposing bound entity-verb-entity triples. Superposition is inherently lossy. Memory degrades as new experiences are added without explicit forgetting rules. The dimension size is a character attribute — high-intelligence NPCs have larger bundles and forget less.

Operations

// All vectors are Int8Array of length GraphInstance.hdcDim (e.g. 8192).
// Binary HDC: values are +1 or -1.

/**
 * Generates a random quasi-orthogonal base vector for an entity or verb type.
 * Each unique id always produces the same vector from the same seed.
 */
function baseVector(id: string, seed: number, dim: number): Int8Array

/**
 * Bind two vectors. Encodes a relationship.
 * bind(A, B) ≠ A, ≠ B, but is retrievable.
 * Implemented as element-wise XOR for binary HDC.
 */
function bind(v1: Int8Array, v2: Int8Array): Int8Array

/**
 * Superpose (bundle) an experience into an existing memory.
 * bundle += bind(bind(sourceVec, verbVec), targetVec)
 * Then re-binarise by sign(). Old memories gradually lose retrieval fidelity.
 */
function remember(bundle: Int8Array, experience: VerbEdge, seed: number): Int8Array

/**
 * Query bundle for the presence of a relationship.
 * Returns cosine similarity in [0, 1]. Above 0.5 indicates likely recall.
 * Not deterministic for distant memories — this is correct behavior.
 */
function recall(bundle: Int8Array, sourceId: string, verbType: string, targetId: string, seed: number): number

Dimension as character attribute

hdcDimApproximate capacitySuggested NPC tier
1024~8 distinct experiences before significant interferenceAmbient, disposable
4096~30–40 experiencesNamed NPCs, minor characters
8192~80–100 experiencesMajor NPCs, faction leaders
16384~200 experiencesExceptional individuals, ancient entities

The capacity estimates above assume experiences of roughly equal salience. High-weight experiences are superposed multiple times (proportional to floor(weight × n)), giving them higher retrieval fidelity at the cost of faster interference buildup for low-weight memories.

06

Multi-Graph Federation

6.1 Graph instance ordering

Graph instances form a directed acyclic dependency graph. An instance may only reference entities in instances that appear earlier in the topological sort. At boot, the system performs this sort and rejects any configuration containing a cycle. The canonical ordering for the three standard instances is:

static-rules → dynamic-world → political

Static-rules references nothing. Dynamic-world may reference entities defined in static-rules. Political may reference entities in both. This ordering is load order: static-rules is always initialized first.

6.2 Boundary verb registry

The boundary verb registry is a whitelist of permitted cross-graph edge types. Any attempted cross-graph write using a verb type not on the whitelist is rejected at commit time.

/**
 * @typedef {Object} BoundaryVerb
 */
{
  verbType:            string,    // Must have MetaVerb.crossGraphAllowed = true.
  sourceGraphId:       string,
  targetGraphId:       string,
  transmissionFidelity:number,    // [0, 1]. See §5.5. Default: 1.0.
  temporalOffset:      number | null,// Log seq delay before target receives. See §6.4.
  directionality:     'one-way' | 'both',
}
Governance note

The boundary verb registry is the primary defence against domain complexity bleed. Adding a boundary verb is a deliberate design decision, not a shortcut. All boundary verb additions require explicit entry in the registry; there is no programmatic way to bypass it.

Boundary verb proliferation (many cross-graph edges accumulating over time) is a signal that two graph instances should be merged or that a third intermediate graph should be introduced.

6.3 Cross-graph ID resolution

A cross-graph reference in a VerbEdge uses a qualified entity reference of the form graphId:entityId. The graph registry resolves these at query time, not at write time. Write-time validation only checks that the target graphId exists and that the verbType is whitelisted for the source→target graph pair.

/**
 * Resolves a possibly-qualified entity reference.
 * If the id has no graphId prefix, the edge's own graphId is assumed.
 */
function resolveEntity(ref: string, context: GraphInstance): EntityNode | null

/**
 * Returns all boundary edges originating from a given entity,
 * along with their resolved foreign targets.
 */
function crossGraphEdges(entityId: string): Array<{edge: VerbEdge, target: EntityNode}>

6.4 Temporal offset — the delay mechanic

A boundary verb with a non-null temporalOffset creates a pending write queue. When the source graph commits an edge of this type, the write is placed in the queue with a deliverAt = current log seq + temporalOffset. The target graph processes the queue and applies the write when the global log seq reaches deliverAt.

/**
 * @typedef {Object} PendingBoundaryWrite
 */
{
  edgePayload:  VerbEdge,
  deliverAt:    number,      // Global log seq number.
  fidelity:     number,      // Copied from BoundaryVerb at enqueue time.
}

The pending queue is persisted as part of the event log. On server restart, undelivered pending writes are replayed from the log and re-queued with their original deliverAt values. No pending write is ever lost.

The temporal offset value is expressed in log sequence units, not wall-clock time. This keeps replay fully deterministic: replaying the event log produces identical pending write delivery regardless of how long the server was running.

07

Query Interface

The query interface exposes four operation families. Fluent DSL syntax is deferred. The following signatures define the underlying operations that any DSL will compile to.

Traversal

/**
 * Traverses the graph from a starting entity along a sequence of verb types.
 * Each hop applies the verb's t-norm to accumulate path weight.
 * Paths with accumulated weight < epsilon are pruned.
 *
 * Returns the set of reachable entities and the path weight to each.
 */
async function traverse(params: {
  startId:    string,
  graphId:    string,
  path:       Array<{ verbType: string, filter?: (edge: VerbEdge) => boolean }>,
  crossGraph: boolean,         // Whether to follow boundary edges. Default: false.
  maxDepth:   number,          // Safety cap. Default: 16.
}): Promise<Map<string, number>>   // entityId → path weight

Scoring

/**
 * Returns the embedding-space score for a (source, verb, target) triple.
 * Does not require an edge to exist. Purely geometric.
 */
function score(sourceId, verbType, targetId, graphId): number   // [0, 1]

/**
 * Returns the fuzzy weight of the best direct edge of verbType
 * from sourceId to targetId, or 0 if none exists.
 */
function edgeWeight(sourceId, verbType, targetId, graphId): number

Nearest-neighbor

/**
 * Returns the k entities whose embeddings are nearest to
 * the vector (source.embedding + verbType.embedding).
 * Entities in a similar relational position to source, via verbType.
 */
async function nearest(params: {
  sourceId:  string,
  verbType:  string,
  graphId:   string,
  k:         number,
  filter?:   (entity: EntityNode) => boolean,
}): Promise<Array<{ entity: EntityNode, score: number }>>

Belief query

/**
 * Returns the BeliefFunction an agent holds about a given proposition.
 * Proposition is a serialised graph pattern string.
 * Returns null if the agent has no belief record for this proposition
 * (not the same as mUnknown = 1.0 — that would be explicit ignorance).
 */
function agentBelief(agentId, proposition, graphId): BeliefFunction | null

/**
 * Returns the objective graph state for a proposition as a fuzzy weight.
 * This is ground truth — what the graph actually contains,
 * not what any agent believes.
 */
function groundTruth(proposition, graphId): number   // [0, 1]

Two-tier query routing

Queries that can be satisfied from the projection cache (L4) should prefer that path for latency-sensitive callers. Queries that require ground-truth graph traversal or embedding computation are routed to the graph core (L2/L3). The query interface is responsible for this routing. Callers declare their latency requirement via a freshness parameter:

freshnessSourceLatencyStaleness risk
'cached'Projection cache only~0.1msUp to one dirty cycle
'graph'Graph core, main thread~1–10msNone
'worker'Graph core, worker pool~10–50msNone, non-blocking
08

Persistence & Replay

The event log is the canonical persistence artifact. All graph state — including inferred edges, embedding updates, and HDC bundle mutations — can be reproduced by replaying the log from a seed. Snapshots are a performance optimization, not a source of truth.

Log flush strategy

Graph instanceFlush triggerFsync policy
static-rulesBoot only (read-only thereafter)Immediate
dynamic-worldEvery N commits or T seconds (configurable)Group fsync per flush
politicalEvery commitImmediate (history must be preserved)

Snapshot checkpoints

A checkpoint records the full serialised graph state at a specific log sequence number. On server start, the engine loads the most recent checkpoint and replays only the log entries after it. Checkpoints are taken automatically when the log exceeds a configurable size threshold. They are also triggered manually before server shutdown.

/**
 * @typedef {Object} Checkpoint
 */
{
  seq:          number,      // Log seq at which the snapshot was taken.
  graphStates:  Map<string, SerializedGraph>,
  pendingQueue: PendingBoundaryWrite[],
  rngState:     string,      // PRNG state at checkpoint time.
  checksum:     string,      // SHA-256 of serialised content.
}

Determinism requirements

All write paths must satisfy the following

No Math.random() or Date.now() in any code path that produces a graph mutation. Use the seeded PRNG exclusively.

No async interleaving within a single graph commit. Each commit is an atomic synchronous operation on the graph data structure, even if the flush to disk is async.

Floating-point operations that affect edge weights or embeddings must use a fixed precision rounding step (toFixed(6)) to prevent drift across replay on different hardware.

External inputs (player actions, NPC decisions) are recorded in the event log before their effects are applied. The log entry is the authoritative record of intent; the graph mutation is its consequence.

Worker pool sizing

A fixed-size worker pool of max(2, os.cpus().length − 1) threads handles all worker-assigned operations. The pool uses a queue with priority lanes: game-critical queries (AI decisions, player-facing results) take priority over background operations (diffusion evaluation, checkpoint writes).

Main-thread budget

The combined cost of a graph commit — validation, log append, embedding update, HDC update, dirty-flag propagation — must complete within 2ms on the target hardware. If profiling indicates this budget is regularly exceeded, the first remediation is reducing embeddingDim. The second is moving embedding updates to a worker with asynchronous reconciliation.

Zone server architecture

Each zone server owns the in-memory portion of the dynamic-world graph for its zone's entities. Cross-zone edges are stored in a shared region of the dynamic-world graph backed by a shared log segment. Entities moving between zones trigger a handoff protocol: the outgoing zone writes an entity:remove event, the incoming zone writes an entity:create event with the same id. The entity's full embedding and HDC bundle travel with the handoff payload.

Hot zone policy

GraphInstance.hotZoneSelector is a function that takes an entity and returns true if it should be retained in memory. The default implementation keeps all entities that have been accessed in the last N minutes (configurable). Cold entities are paged to the backing store. Accessing a cold entity triggers a page-in; the entity is then subject to the hot zone policy on subsequent ticks.

10

Open Questions & Decisions Required

OQ-01   Embedding dimensionality defaults require empirical validation against actual game-scale entity counts. The spec assumes 128 for dynamic-world. This should be confirmed by profiling commit latency with realistic entity density before the embedding layer is finalised.
OQ-02   The HDC dimension table (§5.6) gives approximate capacity estimates. These need validation against actual interference curves using the chosen t-norm and typical experience distributions. The boundary between "ambient NPC" and "named NPC" tiers may shift significantly.
OQ-03   Dempster's rule of combination produces counterintuitive results under high conflict (K near 0). A conflict resolution policy — retain higher authority, blend, or escalate — needs to be selected and registered as a graph-wide default.
OQ-04   The temporal offset in §6.4 is expressed in log sequence units. The relationship between log sequence numbers and in-game time needs to be specified. If the game uses a fixed-tick server, one log seq = one tick is natural. If commits are event-driven, the conversion is less obvious.
OQ-05   Nearest-neighbor queries using a naive O(n × d) scan are acceptable for small graphs but will require an HNSW index at MMORPG scale. The HNSW index must be kept consistent with embedding updates. This is a significant implementation surface that needs its own design pass.
OQ-06   The query interface defines a freshness parameter but does not specify how the projection cache (L4) decides when a dirty component is stale enough to re-derive. The dirty cycle interval needs to be defined per component type, not globally.
OQ-07   The fluent query DSL is deferred. When it is designed, it should compile to the four operation families defined in §7 without adding new runtime primitives. DSL design should happen after at least one integration test of the raw query interface validates the operation shapes.
OQ-08   Floating-point determinism across Node.js versions and hardware is not guaranteed. The toFixed(6) rounding requirement in §8 needs to be stress-tested with a replay harness before being relied upon for save-game correctness.
