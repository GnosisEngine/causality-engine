A few design notes. The dirtyIntervalMs: 0 setting means "always re-derive immediately when dirty" — useful for components that change frequently and where staleness is unacceptable. The default of 1000ms is appropriate for relational components like attitude scores that evolve at human timescales. The flush() method is designed for a scheduled game-loop call every 500ms or so, batching re-derivations rather than firing them on every read.
The HotZone.evict() conservatively refuses to remove entities with connected edges. This prevents orphaned edge references in the store, but it means heavily-connected entities never page out regardless of age. For the MMORPG use case this is correct — removing an entity with edges would corrupt the edge indexes. The caller is responsible for removing edges first if it genuinely needs to evict a connected entity.
