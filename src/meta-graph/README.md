A few things that landed during this phase worth noting. The registry.ts re-stratifies on every registration — not just on inference rule additions but also on verb and taxonomy edge additions, because implies edges compile to rules automatically. This means stratification is always consistent with the full current rule set, at the cost of running the sort more often than strictly necessary. At meta-graph scale (tens to hundreds of verbs, not thousands) that cost is negligible.

The applyStrata function inside the registry has one subtle characteristic: it assigns a verb's inferenceStratum based on the maximum stratum of any rule that derives it. A verb derived by multiple rules in different strata takes the highest. This is conservative but correct — it ensures no rule evaluates a predicate before all rules that could derive it have had a chance to run.

The validator for requiresPolicy edges reuses the factor field to carry the policy id string, which is a mild type abuse. That field was designed for weight multipliers on implies edges. Worth considering whether MetaTaxonomyEdge needs an explicit policyId field in the next spec revision — not blocking, just notable.

One thing to add to the bottom of the existing validateInferenceRule you may want to address: a condition that references the same verb as derivesVerbId is technically valid here but creates a self-referential rule that the stratification pass will never flag as a problem (since it's not a negated cycle). Whether that's a concern depends on whether you want the validator to be purely structural (what it is now) or also catch semantic oddities. Easy to add a check for cond.verbId === rule.derivesVerbId if you want to be explicit about it.
