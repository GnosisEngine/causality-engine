A few design decisions worth noting. The readFrom generator does a full linear scan from the start of the log rather than seeking — this is correct for NDJSON and acceptable given that production replay starts from a checkpoint seq that is close to the end of the log. For a graph with a large log and no checkpointing, this would be expensive, which is exactly why the checkpoint system exists. The replayLog function also intentionally does not re-run the math pipeline during replay — embedding and HDC state is restored from the embedding:update and hdc:update log entries directly. This means replay is as fast as disk read speed, not math computation speed, and produces identical results regardless of PRNG state at replay time.

The batched flush strategy uses a fire-and-forget async write when the batch size is reached. This is appropriate for dynamic-world graphs but means there is a brief window where the buffer holds more than batchSize entries if the async write hasn't completed before the next append. The immediate strategy blocks on datasync before returning, making it safe for political graphs where history preservation is required.
